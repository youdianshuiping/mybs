# mybs 开发规范

本文档定义 mybs 项目的命名、注释、分层职责和重构要求，适用于 `Classes/` 下所有业务代码。

## 1. 命名规范

### 1.1 类名与文件名

- 类名：大写字母开头，采用 PascalCase。
- 文件名：与类名保持一致，大写字母开头。

示例：`GameController` 对应 `GameController.h/.cpp`。

### 1.2 函数名与变量名

- 采用驼峰风格（小写字母开头）。

示例：`startGame`、`stackDrawIndex`。

### 1.3 类的私有成员与私有方法

- 统一以下划线 `_` 开头。

示例：`_model`、`_refreshViewState`。

### 1.4 常量变量名

- 以小写字母 `k` 开头。

示例：`kMaxUndoCount`、`kDefaultAnimationDuration`。

## 2. 代码质量要求

### 2.1 注释要求

- 每个类必须在声明处添加类注释，明确：
  - 功能
  - 职责
  - 使用场景
- 类的成员变量必须添加注释，说明用途。
- 类的公共方法必须添加规范注释，说明参数和返回值。

推荐风格（C++）：

```cpp
/**
 * @brief 启动关卡并绑定视图回调。
 * @param levelId 关卡编号。
 * @param view 游戏视图对象。
 * @return true 表示启动成功，false 表示失败。
 */
bool startGame(int levelId, GameView* view);
```

### 2.2 复杂度与重构阈值

- 当函数代码超过 50 行时，必须进行重构（拆分子函数）。
- 当类代码超过 500 行时，必须进行重构（拆分职责或下沉模块）。

### 2.3 设计原则

- 模块职责明确。
- 遵循单一职责原则（SRP）。
- 优先保证可维护性和可扩展性。

## 3. 分层规范

### 3.1 models 层

- 仅负责数据表达，不包含复杂业务逻辑。
- 必须支持序列化与反序列化（用于存档恢复）。

### 3.2 views 层

- 负责 UI 展示。
- 可持有 `const` 类型的 controller 指针和 model 指针。
- 与其他模块交互优先通过回调接口。

### 3.3 controllers 层

- 负责协调 model 与 view。
- 负责处理用户操作相关业务逻辑。

### 3.4 managers 层

- 主要作为 controller 的成员变量存在。
- 可持有 model 数据并进行加工。
- 禁止实现为单例模式。
- 禁止反向依赖 controller。
- 与其他模块交互优先通过回调接口。
- 可提供全局性的服务和功能。

### 3.5 services 层

- 负责业务逻辑处理，不管理数据生命周期。
- 自身禁止持有数据，但可加工输入数据。
- 可以实现为单例或提供静态方法。

### 3.6 utils 层

- 提供通用独立辅助功能。
- 不涉及业务逻辑，保持完全独立。

## 4. 执行建议

- 新增类时，必须同时满足命名和注释规范。
- 代码评审（CR）时，按本规范逐条检查。
- 对超长函数与超大类，优先在同次需求内完成重构，避免技术债累积。
